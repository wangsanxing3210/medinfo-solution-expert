# 流程示例库（Few-shot Examples）

> 本文件为「医疗信息化解决方案专家 Skill」的**扩展示例库**，与 `SKILL.md` 中的「流程示例（Few-shot）」章节互补。
> 目的：为不同项目场景提供**输入 → 拆解动作 → 关键产出**的示范，降低跨场景产出差异，提升方案编撰稳定性。
> 使用时按需加载，不必全读。完整方案样例见 `assets/sample-solutions.md`。

---

## 一、编撰（Compose）场景示例

> 每个示例给出：①用户输入 ②技能拆解动作 ③关键产出骨架。技能动作中标注了应加载的 references 章节。

### 示例 C1 · 互联互通四级甲等（二级医院）

- **用户输入**："某县人民医院，二级甲等，要做互联互通四级甲等测评，目前 HIS/EMR 在用但没做集成平台，帮我出方案"
- **技能动作**：
  1. 场景=互联互通测评（四级甲等），规模=二级，现状=无集成平台（必须先建）
  2. 加载 `standards.md` 互联互通章：WS/T 798-2022、WS/T 500-2016、WS/T 826-2023
  3. 加载 `scenarios.md` 互联互通场景痛点：共享文档、交互服务、EMPI
  4. 缺口：缺集成引擎、EMPI、53 类共享文档生成器、交互服务
  5. 按模板 12 标题输出
- **关键产出**：摘要+背景+政策依据+现状差距（列 EMPI/共享文档/交互服务三项缺口）+七层架构（重点集成平台+CDR）+详细方案（集成引擎选型、53类文档生成器、交互服务清单）+实施计划（先平台后应用）+风险（数据治理、厂商）

### 示例 C2 · 县域医共体数智一体

- **用户输入**："帮我写一个紧密型县域医共体数智一体建设方案，牵头医院是县人民医院，下面 12 家乡镇卫生院"
- **技能动作**：
  1. 场景=县域医共体数智一体（场景五），规模=1+12，核心=基层下沉+AI 普惠
  2. 加载 `scenarios.md` 场景五：四层一体（数据/业务/智能/运营）、医保协同、云化 SaaS
  3. 加载 `standards.md`：等保三级、WS/T 系列、互联互通
  4. 架构：医共体信息平台 + 集成引擎 + 统一健康档案 + AI 中台下沉
  5. 重点写：双向转诊、检查检验互认、处方流转、远程影像/心电、医保总额打包
- **关键产出**：突出"基层轻量化（云化 SaaS）+ 县级能力中心化"，含 DRG/DIP 医保协同（可联动 `drg-dip.md`）

### 示例 C3 · 城市医疗集团

- **用户输入**："写一份紧密型城市医疗集团数智化方案，三级医院牵头，含多家社区中心，强调网格化管理"
- **技能动作**：
  1. 场景=城市医疗集团（场景八），核心=跨级协同+网格化+医防融合（区别于县域版）
  2. 加载 `scenarios.md` 场景八：多三级协同、网格化管理、DRG/DIP 复杂支付、公卫临床割裂
  3. 架构：集团数据中心 + 网格化运营 + 医防融合平台
  4. 强调：信息系统异构集成（二三级系统不一）、居民跨网格流动、医保多种支付方式
- **关键产出**：区别于县域版的"城市特性"——异构集成策略、网格化运营驾驶舱、医防融合

### 示例 C4 · 中心药房

- **用户输入**："我们医共体要建中心药房，统一采购和配送，给药房方案"
- **技能动作**：
  1. 场景=中心药房（场景六），加载 `central-pharmacy.md` 全库
  2. 业务模型：药品 MDM + 统一采购 + 处方流转 + 药品追溯 + 慢病长处方 + 自动发药
  3. 数据模型：药品主数据 16 字段（对接国家医保统一编码）
  4. 接口：FHIR MedicationRequest 扩展 + 7 类接口
  5. 合规：处方有效期、麻精不流转、保存≥15 年
- **关键产出**：药品主数据治理方案 + 流转链路 + 与云审方的协同（联动 `cloud-prescription.md`）

### 示例 C5 · 云审方中心

- **用户输入**："集团想建云审方中心做前置审方，怎么写方案"
- **技能动作**：
  1. 场景=云审方（场景七），加载 `cloud-prescription.md` 全库
  2. 政策：国卫办医函〔2018〕734 号前置审核强制要求
  3. 架构：规则引擎（Drools 等）+ 药师工作站 + 智能增强（NLP+LLM 辅助）
  4. 规则库：规范/合理/医保三类；规则版本管理（灰度/A/B/回滚）
  5. 指标：响应≤500ms、吞吐 200 处方/分钟
- **关键产出**：规则引擎选型对比 + Drools 规则示例 + 与中心药房的协同数据流

### 示例 C6 · 连锁民营医院

- **用户输入**："连锁口腔集团 8 家门店，要数智化升级，方案怎么写"
- **技能动作**：
  1. 场景=连锁民营（场景九），核心=SaaS 多租户+会员中台+营销 SCRM+合规风控
  2. 加载 `scenarios.md` 场景九：多租户隔离、会员统一、广告合规（医美/口腔高危）
  3. 强调合规：医疗广告法、预付卡合规、病历保存、隐私
  4. 技术：SaaS 化、多门店数据隔离、统一会员 ID
- **关键产出**：多租户架构 + 会员/营销中台 + 合规专章（广告/预付卡/病历）

### 示例 C7 · 三级以下单体机构

- **用户输入**："社区服务中心要上云 HIS+EMR+公卫一体，等保二级，给个轻量方案"
- **技能动作**：
  1. 场景=三级以下单体（场景十），定位=轻量化 SaaS
  2. 分层目标：基础层/标准层/进阶层，避免一刀切
  3. 内容：HIS+EMR+公卫一体化、电子病历 3-4 级、等保二级、医保电子凭证
  4. 架构：云化 SaaS，本地轻量终端
- **关键产出**：轻量云架构 + 公卫临床融合 + 远程协同（对接上级）

### 示例 C8 · DRG/DIP 2.0 专项

- **用户输入**："医院要做 DRG/DIP 2.0 精细化运营，出个方案"
- **技能动作**：
  1. 加载 `drg-dip.md` 全库
  2. 政策：2024-07-23 国家医保局 2.0 通知，2025 起全国统一
  3. 模块：分组器+病案首页质控+医保智能审核+成本核算+绩效（正向）
  4. 硬数据：DRG 核心 409 组/细分 634 组；DIP 核心 9520 组
  5. 红线：禁止将支付标准作限额考核医生；以本地版本为准
- **关键产出**：五位一体运营体系 + 特例单议/除外支付 + 指标（CMI/消耗指数/首页合格率）

## 二、审核（Audit）场景示例

### 示例 A1 · 中心药房方案审核

- **用户输入**："审核这份中心药房建设方案"
- **技能动作**：加载 `audit-checklist.md` 七维度 + `central-pharmacy.md` 合规章
- **重点核查**：
  - ①药品主数据是否对接国家医保统一编码（避免主数据混乱踩坑）
  - ②麻精药品是否明确"不流转"（合规红线）
  - ③处方保存期限≥15 年是否规划
  - ④自动发药机选型是否考虑卡药、追溯码扫描失败
  - ⑤与云审方的协同数据流是否清晰
- **关键产出**：问题清单（✅/⚠️/❌）+ 整改建议 + 优先级

### 示例 A2 · 云审方方案审核

- **用户输入**："看看这份云审方中心方案有没有漏洞"
- **技能动作**：加载 `audit-checklist.md` + `cloud-prescription.md`
- **重点核查**：
  - ①前置审方是否满足 734 号强制要求（事后→事前）
  - ②规则引擎选型是否说明（Drools/OpenL/自研），有无版本管理
  - ③LLM 增强是否标注幻觉风险与人工兜底（AI 专项审核）
  - ④性能指标（≤500ms、200 处方/分钟）是否承诺且有依据
  - ⑤规则误报率是否给出控制策略

## 三、优化（Optimize）场景示例

### 示例 O1 · 智慧医院方案加强数据中台

- **用户输入**："把这份智慧医院方案优化一下，重点加强数据中台和数据流"
- **技能动作**：保留原结构，针对数据流章节补充：
  1. 主数据源归属表（每个数据谁产生、谁拥有、谁消费）
  2. 消息可靠性机制（ACK/重试/死信）
  3. CDR/ODR/SDR 分层与演进路径（避免"中台"一词空泛）
  4. 数据治理独立子项目（与系统建设同步）
- **关键产出**：修改前/后对比 + 具体可执行建议（不重写）

### 示例 O2 · 医共体方案强化医保协同

- **用户输入**："这份县域医共体方案太虚了，医保协同部分帮我做实"
- **技能动作**：
  1. 加载 `drg-dip.md` 医保协同章
  2. 补充：总额打包结算流程、结余留用分配机制、特例单议协同
  3. 上下级病组结构差异 → 资源下沉策略（县级收治复杂病例、基层守慢病）
  4. 给出可量化指标（转诊率、互认率、医保结余率）
- **关键产出**：医保协同从"概念"到"机制+指标"
