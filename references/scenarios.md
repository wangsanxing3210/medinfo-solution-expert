## 场景一：智慧医院建设

### 1. 政策依据

- 《电子病历系统应用水平分级评价标准及评价方法（试行）》
- WS/T 855-2024《医院智慧服务分级评估标准体系》
- WS/T 856-2024《医院智慧管理分级评估标准体系》
- 《"十四五"国民健康规划》
- 国家卫健委"三位一体"智慧医院顶层设计

### 2. 智慧医院三位一体

| 维度 | 核心标准 | 目标等级（三甲参考） |
|------|---------|----------------------|
| 智慧医疗（电子病历分级评价） | 0~8 级 | ≥5 级 |
| 智慧服务 | 0~4 级 | ≥3 级 |
| 智慧管理 | 0~4 级 | ≥2 级 |

### 3. 痛点

- 三大评估体系互不打通，数据采集重复
- 临床决策支持（CDSS）能力薄弱，难达 5 级"中级决策支持"
- 智慧服务外延广，自建成本高
- 智慧管理涉及 HRP 改造，老旧系统改造阻力大

### 4. 方案骨架要点

1. **统一数据底座**：CDR + ODR 一体化建设，支撑三大评估的数据采集
2. **临床决策支持（CDSS）**：规则引擎 + LLM 双引擎，规则覆盖合理用药、危急值、过敏、检查检验互斥；LLM 处理复杂推理
3. **智慧服务门户**：互联网医院 + 自助服务 + AI 导诊，统一患者入口
4. **智慧管理驾驶舱**：BI 大屏 + 智能预警，对接 HRP
5. **数据治理**：主数据管理（患者、人员、物资、字典）、数据质量监控、统一指标口径

### 5. 关键架构图示

```
┌─────────────────────────────────────────────────┐
│ 智慧服务：互联网医院 / 自助机 / 智能导诊 / 随访  │
├─────────────────────────────────────────────────┤
│ 智慧医疗：EMR / HIS / LIS / PACS / 药学 / 手麻 │
├─────────────────────────────────────────────────┤
│ 智慧管理：HRP / 财务 / 物资 / 后勤 / 绩效       │
├─────────────────────────────────────────────────┤
│ 数据中台：CDR + ODR + SDR + 数据治理           │
├─────────────────────────────────────────────────┤
│ AI 中台：CDSS / NLP / 影像 AI / 大模型          │
├─────────────────────────────────────────────────┤
│ 集成平台：集成引擎 + EMPI + 服务总线 + 审计     │
└─────────────────────────────────────────────────┘
```

## 场景二：医院信息互联互通成熟度测评

### 1. 政策依据

- WS/T 798-2022《医院信息互联互通标准化成熟度测评》
- WS/T 790-2021《区域卫生信息互联互通标准化成熟度测评》
- WS/T 500-2016《电子病历共享文档规范》
- WS/T 501-2016 / 502-2016《电子病历与医院信息平台接口 / 交互规范》

### 2. 评价等级

| 等级 | 含义 | 典型适用对象 |
|------|------|--------------|
| 一级 | 数据级 | 已建设完成共享文档，最少要求 |
| 二级 | 文档级 | 数据可以标准文档形式对外发布 |
| 三级 | 服务级 | 平台化、统一 EMPI、对外交互服务 |
| 四级（甲/乙） | 平台级 | 内部集成 + 对外交互闭环 |
| **五级（甲/乙）** | 区域级 | 区域医疗信息共享，三甲医院目标 |

### 3. 五大评价维度（重点）

1. **数据集标准化**（权重高）：53 类共享文档符合 WS/T 500-2016
2. **共享文档标准化**：XML 文档结构符合标准
3. **技术架构情况**：是否建立医院信息平台、集成引擎、EMPI
4. **互联互通交互服务**：是否提供 FHIR / WS/T 502-2016 标准交互
5. **运行性能**：消息延迟、并发、可靠性

### 4. 痛点

- 共享文档生成器自研难度高，需购买成熟中间件
- 老旧系统接口改造工作量大
- 数据质量低导致文档生成错误率高
- 性能测试需通过指定压力测试工具

### 5. 方案骨架要点

1. **平台架构选型**：自研 vs 商业（InterSystems HealthShare、东软、卫宁等）
2. **EMPI 建设**：MedPoint / Initiate / 自研基于规则的匹配合并
3. **53 类共享文档生成器**：购买成熟产品（避免自研）
4. **接口改造清单**：列出所有业务系统的待改造接口
5. **数据质量整改**：字典治理、主数据清洗
6. **性能压测**：消息吞吐、延迟、可靠性指标
7. **申报材料准备**：自评报告 + 实验室评测 + 现场查验

## 场景三：紧密型县域医共体建设（基础版）

> 本场景为基础版，数智化升级版见"场景五"。城市版本见"场景八"。

### 1. 政策依据

- 《紧密型县域医疗卫生共同体建设方案》
- 《紧密型城市医疗集团建设试点工作方案》
- 《关于推进分级诊疗制度建设的指导意见》
- 《"健康中国 2030"规划纲要》

### 2. 业务目标

- **人财物统一管理**：成员单位统一调度、统一财务、统一采购
- **服务同质化**：上下级检查检验互认、双向转诊、远程会诊
- **信息互通**：居民健康档案跨机构调阅、电子处方流转
- **医保总额打包**：医保总额预付、结余留用、合理超支分担

### 3. 典型架构：县域医共体

```
┌──────────────────────────────────────────────────────────┐
│ 县级医院（龙头）  HIS/EMR/LIS/PACS/CDR                  │
├──────────────────────────────────────────────────────────┤
│  ↓↑ 双向转诊   ↓↑ 远程诊断   ↓↑ 处方流转                │
├──────────────────────────────────────────────────────────┤
│ 乡镇卫生院  轻量 HIS + 检验检查申请 + 影像调阅           │
├──────────────────────────────────────────────────────────┤
│  ↓↑ 公卫协同   ↓↑ 健康档案                               │
├──────────────────────────────────────────────────────────┤
│ 村卫生室  全科医生工作站 + 公卫上报                       │
└──────────────────────────────────────────────────────────┘
       ↑ 共享：统一居民健康档案平台（区域平台）
```

### 4. 痛点

- 各级机构系统异构，对接工作量大
- 网络条件参差不齐（村卫生室带宽不足）
- 数据归属与权限边界争议（医保结算数据归属谁）
- 远程诊断业务流程需重新设计（申请 → 接诊 → 报告 → 审核 → 回传）
- 医保打包付费涉及多家医保、商保协同

### 5. 方案骨架要点

1. **统一技术底座**：医共体信息平台 + 集成引擎 + 统一居民健康档案
2. **业务协同场景清单**：双向转诊、远程影像、远程心电、远程病理、远程会诊、检查检验互认、处方流转、统一支付
3. **统一资源管理**：人财物一体化平台（HRP 扩展）
4. **医保协同**：医保结算接口、按病种付费（DRG/DIP）、医共体总额打包结算
5. **网络与终端方案**：乡镇以下采用云化 SaaS，按需提供离线模式
6. **安全合规**：等保 2.0 三级、跨机构数据访问审计、患者隐私同意管理（IHE BPPC）

## 场景四：电子病历系统应用水平分级评价

### 1. 政策依据

- 2025 版《智慧医疗分级评价方法及标准》（已取代国卫办医函〔2018〕479 号；1-8 级、752 项指标、22 个闭环、AI 评价 17 项）
- 《电子病历系统应用水平分级评价标准及评价方法（试行）》（原 479 号体系，历史项目参照）

### 2. 评级目标参考

| 评级 | 关键能力 | 目标对象 |
|------|---------|---------|
| 3 级 | 部门间数据交换 | 二级医院 |
| 4 级 | 全院信息共享，初级决策支持 | 三级医院底线 |
| 5 级 | 统一数据中心，闭环管理，中级决策支持 | 三甲医院主流目标 |
| 6 级 | 全流程闭环，高级决策支持 | 区域龙头医院 |
| 7 级 | 完整电子病历，区域医疗信息共享 | 标杆医院 |

### 3. 评价维度（9 个）

1. 病房医师
2. 病房护士
3. 门诊医师
4. 检查
5. 治疗
6. 药房
7. 病历质量
8. 电子病历基础
9. 电子病历应用

### 4. 关键 5 级要求（核心目标）

- **统一数据中心**：CDR 建设完成
- **数据闭环**：医嘱 → 执行 → 结果 → 回填 → 评价，关键流程闭环
- **中级决策支持**：基于规则的智能提醒（危急值、过敏、药物相互作用）
- **共享文档**：53 类共享文档符合 WS/T 500-2016
- **电子签名**：符合《电子签名法》和医疗电子签名规范

### 5. 痛点

- 老 EMR 系统结构化程度不足，需重构
- CDR 建设周期长，与业务系统并行
- 决策支持规则库覆盖不全
- 数据质量整改难（字典混乱、历史数据脏）
- 评测现场数据抽取工作量大

### 6. 方案骨架要点

1. **EMR 升级**：从文档型 EMR 升级到结构化 EMR
2. **CDR 建设**：统一临床数据视图
3. **决策支持（CDSS）**：规则引擎 + LLM，覆盖用药、检查检验、危急值、过敏
4. **业务流程闭环**：用药闭环（开立 → 配发 → 给药 → 评价）、检验闭环、检查闭环
5. **数据治理**：主数据、字典、数据质量监控
6. **申报准备**：自评报告、材料佐证、模拟测评

## 场景五：紧密型县域医共体数智一体建设

> 在"场景三 紧密型医共体"基础上的数智化升级版，强调"数据底座一体化 + AI 智能化贯穿 + 智能运营驾驶舱"。

### 1. 政策依据

- 《紧密型县域医疗卫生共同体建设方案》
- 《"十四五"国民健康规划》
- 《国务院办公厅关于推进分级诊疗制度建设的指导意见》
- 《关于推进紧密型县域医疗卫生共同体建设监督管理的通知》
- 国家卫健委紧密型县域医共体建设评判标准与监测指标体系

### 2. "数智一体"四层内涵

| 层次 | 内涵 | 关键能力 |
|------|------|---------|
| **数据一体** | 县域统一临床数据中心 + 运营数据中心 | CDR + ODR 跨机构聚合、统一 EMPI、统一主数据 |
| **业务一体** | 人财物统一管理、双向转诊闭环、检查检验互认 | HRP 一体化、转诊平台、互认中心 |
| **智能一体** | AI 贯穿诊疗、运营、监管全链路 | CDSS 下沉基层、智能审方、智能监管驾驶舱 |
| **运营一体** | 医共体智慧运营驾驶舱 | 院长驾驶舱、医保智能审核、绩效分析 |

### 3. 痛点（数智化阶段特有）

- 数据虽互通但未"好用"：基层医生看不到结构化临床视图
- AI 能力集中县级，基层享受不到（影像 AI、CDSS 难下沉）
- 医保打包付费后，缺乏智能审核与风险预警手段
- 县乡村三级运营数据割裂，院长决策靠 Excel
- 信息化重复投入，缺乏统一数智底座

### 4. 方案骨架要点

1. **县域数智底座**：统一云原生平台（K8s + 微服务），县乡村三级 SaaS 化接入，避免重复建设
2. **统一临床数据中心（R-CDR）**：跨机构 FHIR 聚合，提供县域 360 视图
3. **统一运营数据中心（R-ODR）**：人财物 + 医保 + 绩效数据汇聚
4. **AI 中台**：影像 AI、CDSS、智能审方、智能导诊能力中心化，基层按需调用
5. **智能运营驾驶舱**：院长驾驶舱（运营、医保、质量、绩效四维）、监管驾驶舱（合规、风险、舆情）
6. **医保智能审核**：事前提醒 + 事中审核 + 事后分析，对接 DRG/DIP
7. **基层数智赋能**：村卫生室全科医生工作站 + AI 辅助诊断 + 远程会诊一键发起

### 5. 关键架构

```
┌──────────────────────────────────────────────────────────────┐
│ 智能运营层：院长驾驶舱 / 监管驾驶舱 / 医保智能审核           │
├──────────────────────────────────────────────────────────────┤
│ AI 中台：影像 AI / CDSS / 智能审方 / 智能导诊                │
├──────────────────────────────────────────────────────────────┤
│ 数据中台：R-CDR + R-ODR + 主数据 + 数据治理                  │
├──────────────────────────────────────────────────────────────┤
│ 业务系统：县级 HIS/EMR/LIS/PACS + 乡镇轻量 HIS + 村卫生室   │
├──────────────────────────────────────────────────────────────┤
│ 集成平台：集成引擎 + EMPI + 处方流转 + 互认中心              │
├──────────────────────────────────────────────────────────────┤
│ 云原生底座：K8s + 微服务 + 统一身份 + 统一审计                │
└──────────────────────────────────────────────────────────────┘
```

## 场景六：中心药房解决方案

> 医共体 / 医联体 / 医疗集团内部的统一中心药房，承担统一采购、统一配送、处方流转、药品追溯、慢病长处方、自动发药等职能。

### 1. 政策依据

- 《药品管理法》及其实施条例
- 《处方管理办法》（卫生部令第 53 号）
- 《医疗机构药事管理规定》
- 《关于促进"互联网+医疗健康"发展的意见》（处方流转相关）
- 《互联网诊疗管理办法（试行）》
- 《药品信息化追溯体系建设指导意见》
- 国家医保局药品统一编码规则

### 2. 业务目标

| 能力域 | 目标 |
|--------|------|
| 统一采购 | 集中招标、统一目录、统一议价、降低药占比 |
| 统一配送 | 中心药房 → 成员单位 / 患者终端配送 |
| 处方流转 | 处方统一入口 → 审方 → 调剂 → 配送 → 用药跟踪 |
| 药品追溯 | 全链路一物一码追溯（生产 → 流通 → 使用） |
| 慢病管理 | 长处方开具、续方、配送、用药提醒 |
| 自动发药 | 门诊自动发药机、住院单剂量摆药 |

### 3. 痛点

- 各成员单位药品目录不统一，议价能力弱
- 处方流转涉及医院 / 零售药店 / 物流多方协同，流程复杂
- 药品追溯需对接国家药品追溯码平台，技术门槛
- 慢病长处方涉及医保结算、配送、处方有效期管理
- 药学服务前置（审方）能力不足

### 4. 方案骨架要点

1. **药品主数据中心（MDM）**：统一药品字典（对接国家医保统一编码）、统一供应商目录
2. **统一采购平台**：集中招标、合同管理、采购计划、入库验收、对账结算
3. **处方流转平台**：
   - 处方上传（HIS/EMR → 中心药房）
   - 自动审方（规则引擎 + 合理用药知识库）
   - 药师复审（云审方中心，详见 `references/cloud-prescription.md`）
   - 调剂配送（院内药房 / 合作药店 / 第三方物流）
   - 用药跟踪（签收、用药提醒、不良反应上报）
4. **药品追溯**：对接国家药品追溯码平台（码上放心等），入库扫描、出库扫描、终端核验
5. **慢病长处方管理**：慢病目录、长处方模板、续方流程、有效期监控、配送协同
6. **自动发药系统**：门诊自动发药机对接 HIS、住院单剂量摆药机对接 EMR
7. **药学服务**：用药咨询、用药教育、药物警戒（ADR 上报）

### 5. 关键架构

```
┌──────────────────────────────────────────────────────────────┐
│ 患者服务：用药提醒 / 用药咨询 / 配送查询                     │
├──────────────────────────────────────────────────────────────┤
│ 药学服务：云审方 / 用药教育 / ADR 上报                        │
├──────────────────────────────────────────────────────────────┤
│ 中心药房核心：处方流转 / 调剂 / 配送 / 追溯                  │
├──────────────────────────────────────────────────────────────┤
│ 采购管理：统一目录 / 集中招标 / 合同 / 对账                  │
├──────────────────────────────────────────────────────────────┤
│ 主数据：药品字典（医保编码）/ 供应商 / 人员                  │
├──────────────────────────────────────────────────────────────┤
│ 集成：HIS / EMR / 医保 / 国家追溯平台 / 物流                 │
└──────────────────────────────────────────────────────────────┘
```

### 6. 合规要点

- 处方流转须依托互联网医院或实体医院资质
- 处方有效期最长 7 天（普通）/ 12 周（慢病长处方）
- 麻精药品不参与流转，必须院内调配
- 药品追溯码须按国家药监局规范对接

## 场景七：云审方中心解决方案

> 区域处方集中审核中心，为医共体 / 医联体 / 集团内各成员单位提供统一的处方前置审核服务，满足《医疗机构处方审核规范》"药师应当对处方进行前置审核"的强制要求。

### 1. 政策依据

- 《医疗机构处方审核规范》（国卫办医函〔2018〕734 号）
- 《医疗机构药事管理规定》（卫医政发〔2011〕11 号）
- 《处方管理办法》（卫生部令第 53 号）
- 《关于加快药学服务高质量发展的意见》（国卫办医〔2018〕45 号）
- 三级医院评审标准中"药学服务"相关条款

### 2. 业务流程

```
医生开方 → 处方上传云审方中心 → 规则引擎自动审核
                                    ↓
                              [通过] → 返回医生 → 进入调剂
                              [预警] → 药师工作站人工复审
                                    ↓
                              [药师通过] → 返回医生
                              [药师驳回] → 反馈医生修改
```

### 3. 痛点

- 基层医疗机构药师短缺，难以满足前置审核要求
- 各单位审方规则不统一，审核标准不一致
- 规则维护工作量大，临床指南更新频繁
- 药师工作量不均衡，高峰时段审方滞后
- 审方数据未沉淀，缺乏药学质量分析

### 4. 方案骨架要点

1. **统一审方规则库**：
   - 合理用药规则（剂量、频次、给药途径、配伍禁忌、相互作用）
   - 专科规则（儿科、老年、肾肝功能调整）
   - 抗菌药物分级管理规则
   - 麻精药品规则
   - 医保规则（限定支付、诊断药品匹配）
2. **规则引擎**：Drools / 自研，支持规则版本管理、灰度发布、A/B 测试
3. **药师工作站**：处方列表、预警详情、患者上下文（诊断、检验、过敏）、审核操作、反馈模板
4. **智能审方增强**：NLP 解析自由文本处方、LLM 复杂用药推理（试点）
5. **审方质量分析**：审方量、驳回率、规则命中率、科室排名、药学质量报告
6. **知识库管理**：药品知识库、临床指南库、规则编辑器、规则审核流程

### 5. 关键架构

```
┌──────────────────────────────────────────────────────────────┐
│ 药师工作站：复审队列 / 患者上下文 / 反馈模板                  │
├──────────────────────────────────────────────────────────────┤
│ 审方引擎：规则引擎 + NLP 解析 + LLM 推理（试点）              │
├──────────────────────────────────────────────────────────────┤
│ 知识库：合理用药 / 抗菌分级 / 医保规则 / 临床指南             │
├──────────────────────────────────────────────────────────────┤
│ 处方接入：HIS / EMR / 互联网医院 / 中心药房                   │
├──────────────────────────────────────────────────────────────┤
│ 数据底座：CDR 患者上下文 + 药品主数据                          │
└──────────────────────────────────────────────────────────────┘
```

### 6. 合规要点

- 审方规则须经医院药事委员会审核通过
- 审方记录须留存 ≥15 年（病历保存要求）
- 药师须具备相应执业资质
- LLM 推理仅作辅助，最终决策由药师签字

## 场景八：紧密型城市医疗集团数智化建设

> 与"场景三 县域医共体"对应的城市版本，由三级医院牵头，联合二级医院、社区卫生服务中心、康复机构，覆盖城市居民全生命周期健康管理。难点在跨级协同与医保支付协同。

### 1. 政策依据

- 《紧密型城市医疗集团建设试点工作方案》（国卫医函〔2023〕48 号）
- 《关于规范紧密型城市医疗集团建设管理的通知》
- 《"十四五"国民健康规划》
- 《关于推进分级诊疗制度建设的指导意见》
- 紧密型城市医疗集团评判标准与监测指标体系

### 2. 业务目标

- **网格化管理**：按行政区划划分服务网格，集团内医疗机构责任共担
- **服务同质化**：检查检验互认、双向转诊、远程会诊、统一临床路径
- **信息互通**：居民健康档案跨机构调阅、电子处方流转
- **医保协同**：医保总额打包、结余留用、DRG/DIP 协同
- **医防融合**：基本公共卫生服务与临床服务融合

### 3. 与县域医共体的差异

| 维度 | 县域医共体 | 城市医疗集团 |
|------|-----------|-------------|
| 牵头单位 | 县级医院 | 三级综合医院 |
| 成员单位 | 乡镇卫生院、村卫生室 | 二级医院、社区卫生中心、康复机构 |
| 网络条件 | 参差不齐 | 普遍较好，可云化 |
| 三级医院数量 | 通常 1 家 | 可能多家三级协同 |
| 医保支付 | 县域总额打包 | 城市医保复杂，涉及多种支付方式 |
| 居民流动性 | 低 | 高，跨网格就诊常见 |
| 信息化基础 | 薄弱 | 较好，但系统异构严重 |

### 4. 痛点

- 多家三级医院牵头时，谁主导争议大
- 城市居民跨网格就诊频繁，医保结算归属争议
- 二三级医院信息系统异构严重，集成难度高
- 社区卫生服务中心公卫与临床数据割裂
- 医保 DRG/DIP 在集团内协同复杂

### 5. 方案骨架要点

1. **集团统一信息平台**：集成平台 + EMPI + 统一居民健康档案
2. **网格化管理**：按行政区划划分服务网格，签约关系管理
3. **双向转诊平台**：上转 / 下转标准流程、转诊指标监控、转诊后随访
4. **检查检验互认中心**：互认目录、互认规则、互认标识、结果调阅
5. **远程协同中心**：远程影像、远程心电、远程病理、远程会诊
6. **处方流转与云审方**：对接中心药房与云审方中心
7. **医防融合平台**：慢病管理、健康教育、疫苗接种、公卫上报一体化
8. **医保协同**：DRG/DIP 分组协同、总额打包结算、结余留用分配
9. **集团运营驾驶舱**：服务量、转诊率、互认率、医保结余、满意度

### 6. 关键架构

```
┌──────────────────────────────────────────────────────────────┐
│ 集团驾驶舱：运营 / 医保 / 质量 / 转诊 / 互认                 │
├──────────────────────────────────────────────────────────────┤
│ 协同中心：双向转诊 / 远程诊断 / 处方流转 / 互认              │
├──────────────────────────────────────────────────────────────┤
│ 业务系统：三级 HIS/EMR + 二级 + 社区 + 康复                   │
├──────────────────────────────────────────────────────────────┤
│ 数据中台：集团 CDR + ODR + 居民健康档案                       │
├──────────────────────────────────────────────────────────────┤
│ 集成平台：集成引擎 + EMPI + FHIR + 医保接口                   │
└──────────────────────────────────────────────────────────────┘
```

## 场景九：连锁民营医院数智化建设

> 民营医疗集团（口腔、眼科、医美、妇产、康复、中医等专科连锁）的数智化建设，核心是"集团总部管控 + 分院标准化运营 + 会员全生命周期管理 + 营销数字化"。

### 1. 政策依据

- 《医疗机构管理条例》及其实施细则
- 《关于促进社会办医持续健康规范发展的意见》
- 《互联网诊疗管理办法（试行）》
- 《医疗机构门诊质量管理规范（试行）》
- 等保 2.0、《数据安全法》《个人信息保护法》
- 各专科对应的诊疗规范（如口腔、医美、康复等）

### 2. 业务目标

| 能力域 | 目标 |
|--------|------|
| 集团管控 | 财务、人力、采购、品牌、质量统一管控 |
| 分院运营 | 临床、服务、收费标准化，新店快速复制 |
| 会员体系 | 跨院会员一卡通、储值、积分、权益、画像 |
| 营销数字化 | 线上获客、私域运营、转化分析、复购 |
| 供应链 | 耗材、药品、器械统一采购与追溯 |
| 合规风控 | 医疗质量、广告合规、数据安全、舆情 |

### 3. 痛点

- 集团总部与分院系统割裂，数据汇总靠 Excel
- 分院快速扩张时，IT 部署周期长、成本高
- 会员体系割裂，跨院消费体验差
- 营销获客成本高，转化数据不可追溯
- 医疗质量与合规风险（特别是医美、口腔广告合规）
- 耗材成本占比高，缺乏精细化管理

### 4. 方案骨架要点

1. **集团一体化云平台**：SaaS 多租户架构，总部 + 分院统一平台
2. **核心业务系统**：
   - 专科 HIS / EMR（轻量化，符合专科特点）
   - 专科影像（口腔 CBCT、眼科 OCT、医美影像）
   - 收费结算（对接医保 + 商保 + 自费 + 储值）
3. **会员中台**：统一会员 ID、储值账户、积分、权益、画像、跨院消费
4. **营销中台**：私域 SCRM、内容营销、线索管理、转化漏斗、ROI 分析
5. **供应链管理**：耗材采购、库存、消耗追溯、成本核算
6. **财务一体化**：集团财务、分院核算、合并报表、成本分摊
7. **质量与合规**：医疗质量指标、广告合规审核、数据安全审计、舆情监控
8. **BI 决策**：院长驾驶舱、分院对比、会员分析、营销 ROI

### 5. 关键架构

```
┌──────────────────────────────────────────────────────────────┐
│ 集团决策：BI 驾驶舱 / 分院对比 / 营销 ROI                     │
├──────────────────────────────────────────────────────────────┤
│ 营销与会员：SCRM / 会员中台 / 内容营销 / 转化分析             │
├──────────────────────────────────────────────────────────────┤
│ 业务系统：专科 HIS/EMR / 影像 / 收费 / 供应链                  │
├──────────────────────────────────────────────────────────────┤
│ 集团管控：财务 / 人力 / 采购 / 质量 / 合规                    │
├──────────────────────────────────────────────────────────────┤
│ 云平台底座：SaaS 多租户 + 统一身份 + 统一审计                  │
└──────────────────────────────────────────────────────────────┘
```

### 6. 合规要点

- 医疗广告需经卫健部门审查，营销内容须合规
- 会员储值须符合预付卡管理规定
- 患者数据须脱敏后用于营销分析
- 医美等高风险专科须加强知情同意与病历管理
- 等保 2.0 二级（社区分院）/ 三级（集团总部数据中心）

## 场景十：三级以下单体医疗机构数智化建设

> 二级医院、社区卫生服务中心、乡镇卫生院、专科医院等单体医疗机构的数智化建设，预算有限、IT 力量薄弱，核心是"轻量化 SaaS + 云化部署 + 基础达标 + 易运维"。

### 1. 政策依据

- 《医院信息化建设标准与规范》（WS/T 826-2023，按等级适用）
- 《乡镇卫生院信息系统建设基本规范》
- 《社区卫生服务中心信息系统建设规范》
- 电子病历系统应用水平分级评价（目标 3-4 级）
- 等保 2.0 二级（基层机构最低要求）
- 《全国基层医疗卫生机构信息化建设基本规范》

### 2. 业务目标

| 目标层级 | 适用对象 | 关键能力 |
|---------|---------|---------|
| 基础层 | 乡镇卫生院、村卫生室 | 基本 HIS + 公卫上报 + 医保结算 |
| 标准层 | 社区卫生服务中心、二级医院 | HIS + EMR + LIS + 基础电子病历 3 级 |
| 进阶层 | 二级综合医院、专科医院 | HIS + EMR + LIS + PACS + 电子病历 4 级 |

### 3. 痛点

- 预算有限（年信息化预算 ≤50 万的机构占多数）
- IT 人员 1-2 人，无法承担复杂运维
- 系统老旧，多为单机版或 C/S 架构
- 数据安全意识薄弱，等保未达标
- 与上级医院 / 区域平台对接能力不足
- 公卫与临床数据割裂，重复录入

### 4. 方案骨架要点

1. **SaaS 化部署**：核心系统上云，免本地服务器与运维
2. **一体化平台**：HIS + EMR + LIS + 公卫 + 医保结算一体化，避免多系统割裂
3. **基础达标**：
   - 电子病历分级 3-4 级
   - 等保 2.0 二级
   - 医保电子凭证、移动支付
4. **公卫临床融合**：慢病随访、健康教育、疫苗接种与临床数据共享
5. **远程协同**：对接上级医院 / 区域平台，开展远程会诊、远程影像
6. **智能辅助**：基础 CDSS（合理用药）、智能导诊、病案首页智能编码
7. **统一运维**：云平台统一监控、自动备份、远程支持

### 5. 关键架构

```
┌──────────────────────────────────────────────────────────────┐
│ 患者服务：预约挂号 / 智能导诊 / 移动支付 / 报告查询           │
├──────────────────────────────────────────────────────────────┤
│ 临床业务：HIS / EMR / LIS / 影像（轻量）                      │
├──────────────────────────────────────────────────────────────┤
│ 公卫服务：慢病管理 / 健康教育 / 疫苗 / 老人儿童管理          │
├──────────────────────────────────────────────────────────────┤
│ 医保结算：医保电子凭证 / 移动支付 / 商保直赔                 │
├──────────────────────────────────────────────────────────────┤
│ 远程协同：会诊 / 影像 / 心电 / 双向转诊                       │
├──────────────────────────────────────────────────────────────┤
│ SaaS 云底座：多租户 + 统一运维 + 自动备份                     │
└──────────────────────────────────────────────────────────────┘
```

### 6. 合规要点

- 等保 2.0 二级为底线，核心业务系统须达标
- 数据上云须选择通过卫健委认证的医疗云
- 医保接口须符合国家医保局统一规范
- 公卫数据上报须符合国家基本公共卫生服务规范

## 通用方案撰写思路

无论哪种场景，方案结构应遵循以下递进：

1. **政策合规性论证**：列引用的政策与标准
2. **现状评估与差距分析**：从评级目标倒推现状差距
3. **总体架构设计**：技术架构 + 业务架构 + 数据架构 + 安全架构
4. **核心建设内容**：分解为子项目清单
5. **实施路径**：分阶段、分批次，先打底座后建应用
6. **投资估算**：硬件 + 软件 + 集成 + 服务 + 培训
7. **风险与应对**：合规、技术、数据、组织、进度
