# AI 医疗应用场景库

> 本文档收录方案编撰中可融入的 AI 赋能场景。每个场景包含技术成熟度、技术栈、合规风险、典型厂商与接入建议。融入方案前，必须判断"必要性 + 可行性 + 风险可控"三要素。

---

## 一、场景分类总览

| 类别 | 子场景 | 成熟度 | 应用阶段 |
|------|--------|--------|---------|
| **医学影像 AI** | 肺结节、糖网、骨折、乳腺、卒中、冠脉 | 三类证已上市多款 | 临床辅助诊断 |
| **NLP 结构化** | 病历结构化、术语标准化、ICD 编码 | 部分三类证 | 临床辅助 |
| **临床决策支持（CDSS）** | 合理用药、危急值、过敏、检查互斥 | 规则型成熟，LLM 型试点 | 临床辅助 |
| **大模型应用** | 病历摘要、智能问答、临床科研助手 | 内测 / 试点 | 辅助 |
| **智能导诊 / 分诊** | 患者自述症状 → 推荐科室 | 成熟 | 服务优化 |
| **AI 病案质控** | 病历缺陷检测、ICD 编码质控 | 成熟 | 运营管理 |
| **智能审方** | 处方合理性、用药风险 | 规则成熟，AI 增强中 | 临床辅助 |
| **智能随访** | 出院随访、慢病管理 | 成熟 | 服务延伸 |
| **智能导诊机器人** | 院内导航、咨询问答 | 成熟 | 服务优化 |
| **大模型医生工作站** | 病历书写辅助、文书生成 | 试点 | 临床辅助 |

## 二、医学影像 AI 辅助诊断

### 1. 技术栈

- **模型架构**：CNN（ResNet、DenseNet）、U-Net 分割、Transformer（ViT、Swin）、多模态融合
- **训练框架**：PyTorch、TensorFlow、MONAI（医疗影像专用）
- **数据格式**：DICOM 3.0（CT/MR/XR）、DICOM WSI（病理）、DICOM Waveform（心电）
- **推理部署**：NVIDIA Triton Inference Server、ONNX Runtime、TensorRT

### 2. 合规要点（医疗器械软件）

| 注册分类 | 监管文件 | 要求 |
|---------|---------|------|
| **独立软件（SaMD）三类证** | 《医疗器械软件注册技术审查指导原则》《人工智能医用软件产品分类界定指导原则》 | AI 影像诊断辅助需三类证，需提交临床评价、算法性能、网络安全文档 |
| 二类证 | 上述文件 | 部分管理类工具 |
| 一类证 | — | 不直接用于诊断 |

### 3. 风险

- **数据隐私**：训练数据涉患者隐私，需脱敏、伦理审批、人类遗传资源审批（如适用）
- **算法泛化**：跨设备 / 跨医院泛化能力差，需多中心验证
- **可解释性**：医生需理解模型决策，否则不信任
- **责任边界**：AI 仅辅助，最终诊断由医生签字

### 4. 接入方式

- 通过 PACS / RIS 系统嵌入：影像入库 → 触发 AI → 结果以 SR 结构化报告回写
- 不允许 AI 直接修改原始影像数据
- 必须保留 AI 处理日志，符合 ATNA 审计要求

### 5. 典型厂商

- 推想医疗、深透医疗、汇医慧影、依图科技、商汤、联影智能、深睿、鹰瞳科技、数坤科技

## 三、NLP 病历结构化

### 1. 技术栈

- **实体识别（NER）**：BERT / RoBERTa / 医疗领域预训练模型（PCL-MedBERT、MC-BERT）
- **关系抽取**：SpERT、CASREL
- **大模型抽取**：GPT-4、Claude、GLM、DeepSeek、通义千问医疗版，配合 Few-shot Prompting
- **术语标准化**：ICD-10、ICD-9-CM-3、SNOMED CT、医学术语库

### 2. 应用场景

- 入院记录 / 出院小结 / 手术记录自动结构化
- 病案首页自动编码（ICD 主诊断、其他诊断、手术操作）
- 临床科研数据抽取（单病种数据库构建）

### 3. 风险与合规

- **编码准确性**：影响 DRG/DIP 分组，进而影响医保结算，错误代价高
- **人工复核机制**：必须保留医生复核环节
- **数据脱敏**：训练与推理过程中的患者信息脱敏

## 四、临床决策支持系统（CDSS）

### 1. 两类技术路线

#### 路线 A：规则引擎型（成熟）

- **技术**：Drools、OpenL Tablets、自研规则引擎
- **覆盖**：合理用药、过敏、肝肾功能调整、危急值、检查互斥、剂量计算
- **优点**：可解释、可审计
- **缺点**：规则维护工作量大，覆盖不全

#### 路线 B：大模型 + RAG 型（试点）

- **技术**：LLM（GLM-4 / DeepSeek / 通义千问医疗版）+ RAG + 工具调用
- **覆盖**：复杂临床推理、自然语言查询、个性化提醒
- **优点**：覆盖广、灵活
- **缺点**：幻觉风险、推理不可解释、性能成本高
- **建议**：辅助决策，不替代规则引擎

### 2. 推荐架构：双引擎

```
┌─────────────────────────────────────────────┐
│  CDSS 服务层：知识查询、推理、提醒          │
├─────────────────────────────────────────────┤
│ 规则引擎（必选）│ LLM 推理引擎（可选增强）  │
├─────────────────────────────────────────────┤
│  知识库：药品 / 诊断 / 检查检验 / 临床指南  │
├─────────────────────────────────────────────┤
│  CDR：临床数据底座                          │
└─────────────────────────────────────────────┘
```

### 3. 合规要点

- 5 级 EMR 评级要求"中级决策支持"，规则引擎是基础
- 涉及辅助诊断的 CDSS 需医疗器械注册
- LLM 类辅助决策必须保留人工复核痕迹

## 五、智能导诊 / 分诊

### 1. 技术栈

- **意图识别**：BERT 分类 / LLM Prompting
- **科室匹配**：症状-科室知识图谱（自建或购买）+ 协同过滤
- **多轮对话**：LLM 多轮澄清
- **部署形态**：微信小程序、医院 App、自助机、智能音箱

### 2. 关键能力

- 自然语言症状输入（"我胸口闷、左肩疼"）
- 多轮追问（"持续多久？是钝痛还是刺痛？"）
- 推荐科室 + 紧急程度（"建议挂心内科，若疼痛剧烈请立即急诊"）
- 与挂号系统打通，直接预约挂号

### 3. 风险

- **误判风险**：胸痛 / 急腹症等急症误判为普通门诊，需设置急症兜底规则
- **合规边界**：智能导诊不得做出诊断结论
- **数据采集**：症状采集涉及个人敏感信息，需明确告知与同意

## 六、大模型临床科研助手

### 1. 典型应用

- **自然语言查询数据库**：Text2SQL / Text2Pandas
- **文献检索与摘要**：RAG + 知识库
- **科研方案起草**：基于模板 + LLM 润色
- **病例报告生成**：结构化数据 + LLM 生成自然语言摘要

### 2. 技术栈

- **基座模型**：开源（GLM、Qwen、Baichuan、DeepSeek）或商用 API
- **RAG 框架**：LangChain、LlamaIndex、Dify、FastGPT
- **向量库**：Milvus、Weaviate、Qdrant、Chroma
- **知识库**：临床指南库、药品库、医院科研数据库

### 3. 合规要点

- **数据脱敏**：所有送入 LLM 的数据必须脱敏
- **私有化部署**：医疗场景建议私有化部署开源模型
- **审计追溯**：保留 Prompt、模型输出、人工修改记录
- **不直接用于诊断**：仅作为科研辅助

## 七、AI 病案质控

### 1. 技术栈

- **缺陷检测**：NLP 分类（缺失项、矛盾项、时限违规）
- **ICD 编码质控**：编码一致性检查、主诊断准确性
- **规则 + AI 结合**：规则覆盖明确违规项，AI 处理语义级问题

### 2. 典型流程

```
病历提交 → 规则检查（时效、完整性） → NLP 缺陷检测 → ICD 编码质控
       → 缺陷报告 → 反馈医生修改 → 复核 → 归档
```

### 3. 风险

- 病案首页编码错误直接影响 DRG/DIP 结算
- 质控规则需与医院编码员共识，避免误报

---

## 八、智能随访

### 1. 形态

- **外呼机器人**：TTS + ASR + 对话管理
- **短信 / 微信推送**：模板化随访问卷
- **慢病管理平台**：可穿戴设备 + 数据采集 + 风险预警

### 2. 合规

- 涉及互联网诊疗，需具备互联网医院资质
- 随访内容不得做出诊断结论，仅收集与提示
- 电话外呼需遵守《通信短信息服务管理规定》

---

## 九、AI 融入方案的关键评估问题

融入方案前，对每个 AI 应用回答以下问题：

1. **必要性**：是否解决具体业务痛点？ROI 是否合理？
2. **数据底座**：训练 / 推理所需要的数据是否齐备？数据质量如何？
3. **合规性**：是否需要医疗器械注册证？涉及哪类数据？
4. **可解释性**：医生是否信任？如何解释？
5. **风险兜底**：AI 失效时的兜底机制是什么？
6. **运维成本**：模型迭代、规则更新、数据治理的成本
7. **人机协同**：人工介入点在哪里？AI 与医生的责任边界？

如果以上问题中有 2 项以上回答不清晰，应在方案中明确标注为"试点 / 探索"而非"全面部署"。
