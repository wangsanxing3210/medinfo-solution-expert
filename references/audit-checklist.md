# 方案审核检查清单

> 本清单用于对已有解决方案进行逐项审核。审核时按"政策符合性 → 逻辑完整性 → 技术可行性 → 数据流闭环 → 安全合规 → 风险应对 → 可落地性"七个维度逐项检查。每项检查后给出结论：✅ 通过 / ⚠️ 待优化 / ❌ 不通过，并附修改建议。

---

## 一、政策符合性审核

### 1.1 政策依据充分性

- [ ] **是否引用最新政策版本**：检查引用的所有政策、标准是否为最新有效版本（特别注意 2023 年后修订的标准，如 WS/T 826-2023、WS/T 855-2024、WS/T 856-2024；**以及 2025 版《智慧医疗分级评价方法及标准》已取代 2018 版 479 号**）
- [ ] **政策引用格式规范**：是否使用「标准编号 + 标准名称」标准格式
- [ ] **政策范围匹配**：引用政策是否覆盖项目实际范围（医政、医保、药监、卫健、网信、公安等）
- [ ] **政策时效性**：是否覆盖最新"十四五"规划与近 2 年新发文件
- [ ] **地方政策补充**：是否补充地方配套政策（如本省智慧医院建设方案）

### 1.2 等级 / 评级目标合规

- [ ] **评级目标合理**：评级目标是否符合医院实际（如三级医院智慧医疗分级 ≥5 级，互联互通 ≥4 级甲等）
- [ ] **多评级协同**：是否考虑智慧医疗、互联互通、智慧服务、智慧管理四者的协同
- [ ] **关键节点对应**：方案是否覆盖评级关键节点的具体要求（如 5 级智慧医疗的"中级决策支持""统一数据中心""22 个闭环管理"）

### 1.3 数据安全合规

- [ ] **等保级别匹配**：核心业务系统等保级别是否符合《医疗卫生机构网络安全管理办法》（二级或三级）
- [ ] **数据出境合规**：如涉及数据出境，是否符合《数据出境安全评估办法》
- [ ] **个人信息保护**：是否单独取得敏感个人信息（医疗健康）处理同意
- [ ] **人类遗传资源**：如涉及人类遗传资源，是否符合《人类遗传资源管理条例》

---

## 二、逻辑完整性审核

### 2.1 章节结构完整性

- [ ] **必备章节齐全**：是否包含摘要、需求分析、总体架构、技术方案、实施计划、风险应对六大章节
- [ ] **章节递进合理**：从需求 → 架构 → 技术 → 实施的逻辑链是否通畅
- [ ] **章节详略得当**：核心技术章节详写，背景介绍简写

### 2.2 需求-方案对应

- [ ] **需求可追溯**：每条业务需求是否在方案中可找到对应解决措施
- [ ] **方案不超范围**：方案是否引入了需求未提到的内容（"超售"风险）
- [ ] **需求优先级**：是否区分核心需求与可选需求

### 2.3 论证逻辑

- [ ] **痛点 → 方案推导清晰**：是否从痛点直接推出方案，而非"先有方案再贴痛点"
- [ ] **方案 → 收益量化**：方案承诺的收益是否有量化指标（如效率提升 30%、纸质流程减少 80%）
- [ ] **量化指标可衡量**：指标是否可被后期验收衡量

---

## 三、技术可行性审核

### 3.1 架构合理性

- [ ] **分层清晰**：业务架构 / 数据架构 / 应用架构 / 技术架构分层是否清晰
- [ ] **集成方式合理**：是否使用集成平台 + 标准接口（HL7/FHIR/IHE），避免点对点接口爆炸
- [ ] **EMPI 建设**：是否建设统一患者主索引
- [ ] **数据中台位置**：CDR/ODR/SDR 是否分层合理
- [ ] **云化 vs 本地化**：是否合理选择云化、本地化、混合部署

### 3.2 技术选型

- [ ] **成熟度**：所选技术栈是否成熟、有规模化案例
- [ ] **可演进性**：是否考虑未来 3-5 年技术演进
- [ ] **国产化**：是否考虑信创要求（信创目录内的数据库、中间件、操作系统）
- [ ] **避免厂商绑定**：是否避免过度厂商绑定
- [ ] **AI 选型合理**：AI 模型选型是否匹配业务场景（影像 CNN/Transformer、NLP LLM、CDSS 双引擎）

### 3.3 性能与容量

- [ ] **并发估算**：是否估算峰值并发（挂号、检查、缴费高峰）
- [ ] **存储容量估算**：影像年增量估算、数据库年增量估算
- [ ] **网络带宽**：跨机构、互联网带宽是否充足
- [ ] **响应时间**：核心交易响应时间是否达标（如挂号 <2s、报告查询 <3s）

### 3.4 高可用与容灾

- [ ] **高可用设计**：是否双机热备 / 集群
- [ ] **数据备份**：是否满足 RPO/RTO（核心业务 RPO ≤15min）
- [ ] **异地容灾**：核心数据是否异地容灾
- [ ] **故障切换**：是否设计自动故障切换

---

## 四、数据流闭环审核

### 4.1 业务闭环

- [ ] **门诊闭环**：挂号 → 就诊 → 检查检验 → 取药 → 结算，全流程数据贯通
- [ ] **住院闭环**：入院 → 医嘱 → 执行 → 结果 → 出院，全流程数据贯通
- [ ] **用药闭环**：开立 → 审方 → 配发 → 给药 → 评价，全流程数据闭环（5 级 EMR 关键要求）
- [ ] **检验闭环**：申请 → 采样 → 标本 → 检测 → 报告 → 回填 → 评价
- [ ] **检查闭环**：申请 → 预约 → 检查 → 报告 → 审核 → 回填

### 4.2 跨系统数据流

- [ ] **数据流向明确**：每个系统的输入输出数据流是否画清
- [ ] **数据所有权**：每条数据的"主数据源"是否明确（如患者信息以 HIS 为主、临床文书以 EMR 为主）
- [ ] **数据一致性问题**：跨系统数据不一致时的处理机制（如以谁为准）
- [ ] **消息可靠性**：集成平台消息的可靠性保障（重试、死信、幂等）

### 4.3 跨机构数据流

- [ ] **跨机构共享标准**：是否使用 WS/T 500-2016 共享文档 / FHIR
- [ ] **患者隐私同意**：跨机构调阅是否经过患者授权（IHE BPPC）
- [ ] **审计追溯**：跨机构数据访问是否记录审计日志

---

## 五、安全合规审核

### 5.1 等保合规与商用密码

- [ ] **等保级别确定**：明确各系统的等保级别
- [ ] **等保技术要求**：物理、网络、主机、应用、数据五层防护是否到位
- [ ] **等保测评周期**：是否规划定期测评
- [ ] **密评（商用密码应用安全性评估）**：等保三级及以上系统是否已规划/通过密评（依据《密码法》27 条、《商用密码管理条例》41 条、GB/T 39786-2021）；《三级医院评审标准（2025版）》已将密评设为前置否决项
- [ ] **国密算法应用**：高级别系统（5 级起）是否采用国密 SM2/SM3/SM4、国密 TLS 传输加密，替换存在风险的国际算法

### 5.2 数据安全

- [ ] **数据分类分级**：是否按《医疗健康数据安全指南》对数据分类分级
- [ ] **国密加密**：是否采用国密算法（SM2/SM3/SM4）进行存储与传输加密，而非仅国际算法
- [ ] **加密传输**：核心数据传输是否使用国密 TLS 1.3 / TLS 1.2+
- [ ] **加密存储**：敏感数据存储是否加密（含字段级加密）
- [ ] **脱敏访问**：测试 / 科研 / 数据分析场景是否脱敏
- [ ] **数据访问审计**：是否记录所有敏感数据访问
- [ ] **全生命周期保护**：是否落实《医疗卫生机构数据安全和个人信息保护管理办法（试行）》（国卫规划发〔2026〕6 号）要求的加密/鉴权/脱敏/去标识化/数字水印

### 5.3 应用安全

- [ ] **身份认证**：是否支持统一身份认证（SSO）、多因素认证
- [ ] **权限控制**：是否 RBAC + ABAC，按角色与属性细粒度授权
- [ ] **越权防护**：医生越权访问其他科室患者数据的防护机制
- [ ] **接口安全**：API 网关、Token、签名防重放
- [ ] **日志审计**：是否符合 ATNA / WS/T 500-2016 审计要求

### 5.4 终端安全

- [ ] **医生工作站**：是否禁止 USB、限制软件安装
- [ ] **移动设备**：移动查房设备的 MDM 管理
- [ ] **自助机**：自助机的物理防护与系统加固

---

## 六、风险应对审核

### 6.1 项目风险

- [ ] **进度风险**：是否识别工期紧张风险及应对
- [ ] **预算风险**：是否识别预算超支风险及应对
- [ ] **关键人员风险**：是否识别关键人员流失风险
- [ ] **厂商风险**：是否识别厂商交付能力、服务延续性风险

### 6.2 业务风险

- [ ] **业务中断风险**：上线期间业务中断的应对
- [ ] **数据迁移风险**：历史数据迁移失败的应对
- [ ] **使用推广风险**：医生 / 患者使用率不达预期的应对
- [ ] **合规变更风险**：政策标准变更的应对

### 6.3 技术风险

- [ ] **技术演进风险**：技术栈过时的应对
- [ ] **集成失败风险**：与老旧系统集成失败的应对
- [ ] **性能不达标风险**：上线后性能不达标的应对
- [ ] **数据质量风险**：数据治理失败的应对

### 6.4 风险描述质量

- [ ] **风险描述具体**：是否避免"可能存在风险"等空泛描述
- [ ] **风险等级评估**：是否评估风险等级（高 / 中 / 低）
- [ ] **应对措施可行**：应对措施是否具体可执行
- [ ] **责任主体明确**：每项风险的责任主体是否明确

---

## 七、可落地性审核

### 7.1 实施计划

- [ ] **阶段划分合理**：是否分阶段、分批次实施
- [ ] **里程碑明确**：是否设置清晰里程碑
- [ ] **依赖关系清楚**：子项目间依赖关系是否清楚
- [ ] **资源配置明确**：人员、设备、网络、资金配置是否明确

### 7.2 组织保障

- [ ] **项目组织架构**：是否成立项目领导小组 / 工作小组
- [ ] **业务部门参与**：业务部门（医务、护理、药学、财务）是否纳入
- [ ] **培训计划**：是否覆盖培训对象、内容、形式、考核

### 7.3 投资估算

- [ ] **预算项完整**：硬件 / 软件 / 集成 / 服务 / 培训 / 运维是否齐全
- [ ] **预算合理**：单价是否符合市场行情
- [ ] **运维成本**：3-5 年运维成本是否纳入
- [ ] **隐性成本**：是否考虑数据治理、培训人力等隐性成本

### 7.4 验收标准

- [ ] **验收指标可衡量**：是否给出可量化的验收指标
- [ ] **验收方法明确**：是否说明验收方法（功能测试 / 性能测试 / 现场查验）
- [ ] **质保期约定**：是否约定质保期与服务响应

---

## 八、AI 应用专项审核（如方案含 AI）

- [ ] **AI 必要性**：每个 AI 应用是否解决具体业务痛点
- [ ] **数据底座**：AI 训练 / 推理所需要的数据是否齐备
- [ ] **合规性**：是否需要医疗器械注册证？是否取得？
- [ ] **可解释性**：是否考虑 AI 决策可解释性
- [ ] **风险兜底**：AI 失效时的人工兜底机制
- [ ] **人机协同**：AI 与医生的协作流程是否清晰
- [ ] **迭代成本**：模型迭代、规则更新成本是否估算
- [ ] **伦理审查**：是否经过医院伦理委员会审查
- [ ] **数据脱敏**：训练数据是否脱敏、是否取得患者授权

## 九、方案预评分与自检报告（编撰后自动自检）

> 本节约用于「流程四 · 方案自检与预评分」：在方案编撰（流程一）完成后自动对生成方案跑一遍本章审核清单，输出加权预评分与等第，快速判定"能否提交"。亦可对任意已存在方案按需触发（用户输入"自检 / 预评分 / 跑一遍审核清单 / 检查方案质量"）。

### 9.1 预评分维度与权重

| 维度 | 权重 | 对应章节 |
|------|------|---------|
| 政策符合性 | 20% | 一 |
| 安全合规 | 20% | 五 |
| 技术可行性 | 15% | 三 |
| 数据流闭环 | 15% | 四 |
| 逻辑完整性 | 12% | 二 |
| 可落地性 | 10% | 七 |
| 风险应对 | 8% | 六 |

- 合计 100%。
- **AI 专项（第八章）**：若方案含 AI 应用，单独计「AI 合规诊断分」（0-100），**不计入总分权重**，但须在报告中单列；诊断分 < 60 视为高危项，整改优先级置顶。

### 9.2 单维度计分

对每个维度，逐项判定：

- ✅ 完全满足
- ⚠️ 部分满足 / 待优化（按建议整改后可满足）
- ❌ 不满足（必须整改）

单维度得分 = (✅ 项数 × 1.0 + ⚠️ 项数 × 0.5) / 该维度检查项总数 × 100，四舍五入取整。

### 9.3 总分与等第

- 总分 = Σ(维度得分 × 权重)，四舍五入取整（0-100）。
- 等第：
  - **≥90 优秀**：可直接提交 / 进入下一环节
  - **75-89 良好**：存在少量 ⚠️，小改即可
  - **60-74 合格**：存在 ❌ 或较多 ⚠️，需针对性修改
  - **<60 不通过**：结构性问题，建议重写相关章节
