# 医疗信息化标准规范参考库

> 本文档收录方案编撰与审核中高频引用的核心标准。引用时一律使用标准编号 + 标准名称的标准格式（例：GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》）。所有标准以最新有效版本为准。

## 一、数据交换与互操作性标准

### HL7（Health Level Seven）

| 标准族 | 用途 | 在国内医疗方案中的应用 |
|--------|------|----------------------|
| **HL7 V2.x** | 医疗系统间消息交换 | HIS/LIS/PACS 等系统间接口事实标准，国内项目仍大量使用 V2.5 |
| **HL7 V3 / CDA R2** | 临床文档结构化 | 电子病历共享文档、健康档案共享规范基础 |
| **HL7 FHIR R4** | 现代 RESTful 资源模型 | 互联互通、互联网诊疗、医联体跨机构数据共享首选；新版互联互通测评要求支持 FHIR |
| **HL7 CCOW** | 临床上下文管理 | 多系统集成时统一患者上下文 |

**方案撰写要点**：
- 跨机构 / 互联互通场景默认采用 FHIR R4，老院区内部遗留接口 V2.x 保留
- 引用 FHIR 资源时点名具体 Resource（Patient、Encounter、Observation、DiagnosticReport、MedicationRequest、DocumentReference 等）
- 病历共享文档遵循 CDA 模板衍生出的 WS/T 500-2016、WS 445-2014

### IHE（Integrating the Healthcare Enterprise）

IHE 是 ITI/Radiology/Cardiology/Patient Care Coordination 等领域的集成用例框架，方案中应优先点名 Profile：

| Profile | 全称 | 应用场景 |
|---------|------|---------|
| **PIX/PDQ** | Patient Identifier Cross-Referencing / Patient Demographics Query | 患者主索引 EMPI 与跨系统患者匹配 |
| **XDS/XDS-I** | Cross-Enterprise Document Sharing (含影像) | 跨机构文档/影像共享、医联体影像调阅 |
| **XCA** | Cross-Community Access | 跨社区 / 跨医联体的文档检索 |
| **ATNA** | Audit Trail and Node Authentication | 医疗系统审计日志、安全审计 |
| **CT** | Consistent Time | 跨系统时间同步 |
| **SWF** | Scheduled Workflow | 放射 / 检查预约到报告的标准化流转 |
| **PCC** | Patient Care Coordination | 急诊、转诊、出院摘要等临床协作 |
| **BPPC** | Basic Patient Privacy Consents | 患者隐私同意管理 |

**方案撰写要点**：在描述集成平台或跨机构数据交换时，用 IHE Profile 表达集成模式，比"采用 HL7"更精确、更专业。

## 二、国内医疗信息化核心标准（卫健委 / 国标）

### 1. 信息安全类（公安部等保 / 国标）

| 标准编号 | 标准名称 | 关键要求 |
|----------|---------|---------|
| GB/T 22239-2019 | 《信息安全技术 网络安全等级保护基本要求》 | 等保 2.0 基本要求框架：物理/网络/主机/应用/数据/管理 + 云/移动/物联网/工控扩展 |
| GB/T 25070-2019 | 《信息安全技术 网络安全等级保护安全设计技术要求》 | 等保安全设计 |
| GB/T 28448-2019 | 《信息安全技术 网络安全等级保护测评要求》 | 等保测评方法 |
| GB/T 39786-2021 | 《信息安全技术 信息系统密码应用基本要求》 | 密评（商用密码应用安全性评估）核心依据；物理/网络/设备/应用/数据四层密码应用要求 |

**等保级别参考**：
- 二级：一般医院业务系统（HIS、EMR、LIS、PACS）
- 三级：三级医院核心业务系统、互联网医院、区域健康平台
- 四级：省级及以上全民健康信息平台

**商用密码与密评（2025 版起刚性要求）**：
- 依据《密码法》第二十七条、《商用密码管理条例》第四十一条：关键信息基础设施、等保三级及以上系统 **必须使用商用密码保护，并定期开展商用密码应用安全性评估（密评）**
- 密评依据 GB/T 39786-2021，覆盖物理环境、网络通信、设备计算、应用数据四个维度；未通过密评不得投入运行，运行后每年至少评估 1 次
- **《三级医院评审标准（2025版）》已将密评结果设为信息化"前置否决项"**——未通过密评的系统无法参与评审数据采集，直接影响三甲资质申报
- 国密算法（SM2/SM3/SM4）在高级别系统（5 级起）强制要求；数据传输加密采用国密 TLS 1.3

### 3. 电子病历 / 智慧医疗分级评价

> ⚠️ **标准已升级**：原《电子病历系统应用水平分级评价管理办法（试行）》（国卫办医函〔2018〕479 号）已被 **《智慧医疗分级评价方法及标准（2025版）》** 全面取代。2025 版由"电子病历分级评价"升级为"智慧医疗分级评价"，是"三位一体"智慧医院建设中"智慧医疗"维度的核心评价标准。

- **名称变更**：电子病历系统应用水平分级评价 → 智慧医疗分级评价
- **级别调整**：取消 0 级，调整为 **1-8 级**（共 8 级）
- **角色调整**：评价角色共 12 个，新增 **医疗质量管理**（含质控、准入、感控、不良事件管理）、**电子病历安全**（强化"人防+技防"体系）2 个角色
- **指标调整**：评价指标由 779 项精简至 752 项；新增项目 8 项、合并减少 6 项、删除 3 项，项目总数 38 项
- **关键新增要求**：
  - **22 个闭环场景**：医嘱、检查、检验、药品、会诊、不良事件、治疗等全流程闭环管理（五级起重点要求）
  - **区域协同**：从 **5 级**开始要求区域数据使用（查询→利用→健康自采），涉及 55 项
  - **人工智能 17 项**：方案推荐、辅助判断、病历辅助生成、病历内涵质控、数字疗法（院后诊疗方案、自采数据形成治疗计划）等，基本为 7-8 级高级别要求
  - **专科要求**：6 级实现 1 个专科、7 级实现 3 个以上专科
  - **国产替代**：OFD、信创等要求；时间源北斗、加密采用国密算法
  - **安全要求下沉**：安全要求从 5-6 级降为 3-4 级，增加个人信息保护、数据安全要求
- **计算方法**：赋分制，按角色分别评价，得分 > 总分 80% 通过
- **数据质量维度**：由 4 个调整为 5 个（唯一性、完整性、合规性、同一性、时效性）

关键节点参考：
  - **4 级**：部门间数据交换，全院信息共享，初级医疗决策支持
  - **5 级**：统一数据中心，闭环管理，中级决策支持（多数三甲医院目标）
  - **6 级**：全流程闭环，高级决策支持，完整电子病历（单体医院可达最高级）
  - **7-8 级**：区域医疗信息共享、临床科研一体化、AI/数字疗法高级应用

> 📌 **政策趋势提示（编撰时务必用 WebSearch 核验）**：2025 版已显现电子病历、互联互通、智慧服务、智慧管理四大体系整合方向，业内预期 2027 年前后将进一步整合为统一的"数智医院"评价标准；同时医保 DRG/DIP 2.0、密评前置等要求正深度耦合评级。具体文号与生效时间以国家卫健委最新发布为准，禁止写死未证实的具体文号。

### 4. 互联互通成熟度测评

依据 WS/T 798-2022《医院信息互联互通标准化成熟度测评》：
- 等级：一级至五级（甲、乙）
- 评价维度：
  - **数据集标准化**（占比较大）
  - **共享文档标准化**
  - **技术架构情况**
  - **互联互通交互服务**
  - **运行性能**
- 五级甲等是三甲医院 / 区域龙头医院的目标，要求数据标准化、平台化、统一患者主索引、对外交互服务等全方位到位

### 5. 智慧医院评估三位一体

| 维度 | 评估文件 | 评价重点 |
|------|---------|---------|
| **智慧医疗** | 《智慧医疗分级评价方法及标准（2025版）》 | 临床诊疗信息化、决策支持、AI/数字疗法 |

## 三、方案引用规范的写法

在方案中引用标准时，遵循下列统一格式：

> 依据 **GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》** 第 X 章 Y 节要求，本系统应实现 ZZZ……

首次出现使用完整编号 + 名称，再次出现可简化为「GB/T 22239-2019」。
