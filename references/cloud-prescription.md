## 一、审方业务模型详解

### 1.1 全链路业务流程

```
医生开方（HIS/EMR/互联网医院）
    ↓ 处方上传（FHIR MedicationRequest + 患者上下文）
云审方中心接收
    ↓
【第一道：规则引擎自动审核】
    ├─ ✅ 全部通过 → 返回医生 → 进入调剂
    ├─ ⚠️ 预警（疑似问题） → 进入药师工作站复审
    └─ ❌ 硬规则拦截 → 直接驳回 → 反馈医生修改
    ↓
【第二道：药师工作站人工复审】
    ├─ ✅ 药师通过 → 返回医生 → 进入调剂
    ├─ ⚠️ 药师有条件通过（注明注意事项）→ 返回医生
    └─ ❌ 药师驳回 → 反馈医生修改原因
    ↓
审方结果回传 + 审方记录归档
    ↓
药学质量分析（月度报告、科室排名、规则优化）
```

### 1.2 处方审核内容分类

| 类别 | 审核项 | 典型规则示例 |
|------|--------|-------------|
| **规范性审核** | 处方完整性 | 科室、诊断、用法用量必填 |
| | 处方权限 | 抗菌分级管理、毒麻精药品权限 |
| | 处方有效期 | 普通 7 天、急诊 1 天、长处方 12 周 |
| **合理性审核** | 适应症 | 诊断与药品适应症匹配 |
| | 用法用量 | 单次剂量、日剂量、频次、疗程 |
| | 给药途径 | 口服/静脉/肌注/吸入等合理性 |
| | 联合用药 | 配伍禁忌、相互作用、重复用药 |
| | 特殊人群 | 儿童、孕妇、老人、肝肾功能调整 |
| | 抗菌药物 | 分级管理、特殊使用级审批 |
| **医保规则** | 限定支付 | 医保目录限定支付条件 |
| | 诊断药品匹配 | 诊断与医保支付药品匹配 |
| | 集采品种优先 | 同类药品集采品种优先 |
| | 自费告知 | 自费药品须患者知情同意 |

### 1.3 药师工作站核心功能

| 模块 | 功能 |
|------|------|
| **复审队列** | 待审处方列表、优先级排序、超时预警 |
| **处方详情** | 处方基本信息、药品列表、用法用量 |
| **患者上下文** | 诊断、过敏史、检验检查、肝肾功能、既往用药 |

## 二、规则引擎选型

### 2.1 主流方案对比

| 方案 | 类型 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **Drools** | 开源规则引擎 | 成熟、社区活跃、支持 RETE 算法、规则 DSL | 学习曲线陡、复杂规则维护难 | 中大型机构、有研发能力 |
| **OpenL Tablets** | 开源 | 规则维护基于 Excel，业务友好 | 性能略逊 Drools、社区较小 | 业务规则频繁变更场景 |
| **IBM ODM** | 商业 | 成熟、可视化规则编辑、版本管理完善 | 价格高、国产化适配弱 | 大型三甲、外资背景 |
| **自研规则引擎** | 自研 | 完全可控、深度定制 | 研发成本高、成熟度风险 | 有强大研发团队的厂商 |

### 2.2 选型建议

- **推荐**：Drools（中小机构）+ 自研规则编辑器（基于 Web 的可视化编辑）
- **大机构**：Drools + 商业合理用药知识库（美康 / 大通）
- **避免**：纯自研规则引擎，除非团队规模 ≥10 人且长期投入

### 2.3 规则版本管理与发布

| 能力 | 要求 |
|------|------|
| 规则版本化 | 每次修改生成版本号、可回溯 |
| 灰度发布 | 新规则先在试点科室灰度，再全院推广 |
| A/B 测试 | 同一规则多版本并行，对比效果 |
| 紧急回滚 | 出问题时一键回滚至上版本 |
| 审批流程 | 规则修改须经药事委员会审批 |
| 生效时间 | 支持定时生效、定时失效 |

### 3.3 规则模板示例

#### 规则模板 1：剂量审核

```drl
rule "剂量审核-成人单次剂量超限"
when
    $rx : MedicationRequest(
        patient.age >= 18,
        $dose : dosageInstruction[0].doseAndRate[0].doseQuantity.value,
        $med : medicationCodeableConcept.coding[0].code,
        $dose > getMaxSingleDose($med, patient.age, patient.weight)
    )
then
    insert(new ReviewWarning(
        WarningLevel.WARN,
        "单次剂量超限",
        "药品 " + $med + " 单次剂量 " + $dose + " 超过推荐最大剂量",
        "Ref: 药品说明书"
    ));
end
```

#### 规则模板 2：抗菌药物分级

```drl
rule "抗菌药物分级-特殊使用级权限审核"
when
    $rx : MedicationRequest(
        medicationCodeableConcept.coding[0].code in SPECIAL_USE_ANTIBIOTICS,
        requester.antibioticLevel < AntibioticLevel.SPECIAL_USE
    )
then
    insert(new ReviewWarning(
        WarningLevel.REJECT,
        "抗菌药物权限不足",
        "特殊使用级抗菌药物须副主任医师以上权限",
        "Ref: 抗菌药物临床应用管理办法"
    ));
end
```

#### 规则模板 3：配伍禁忌

```drl
rule "配伍禁忌-静脉配伍"
when
    $rx1 : MedicationRequest(
        dosageInstruction[0].route.coding[0].code == "IV",
        $med1 : medicationCodeableConcept.coding[0].code
    )
    $rx2 : MedicationRequest(
        dosageInstruction[0].route.coding[0].code == "IV",
        $med2 : medicationCodeableConcept.coding[0].code,
        this != $rx1,
        isIncompatibility($med1, $med2)
    )
then
    insert(new ReviewWarning(
        WarningLevel.WARN,
        "静脉配伍禁忌",
        "药品 " + $med1 + " 与 " + $med2 + " 存在静脉配伍禁忌",
        "Ref: 静脉配伍禁忌表"
    ));
end
```

## 四、智能审方增强

### 4.1 NLP 解析自由文本处方

- **场景**：医生在"用法"栏写"口服，饭后，每次2片，每日3次"
- **技术**：BERT / RoBERTa 命名实体识别 + 规则解析
- **输出**：结构化字段（frequency=3、period=1d、route=PO、dose=2、unit=片、note=饭后）

### 4.2 LLM 复杂推理（试点）

- **场景**：多药联用、复杂患者、罕见情况规则未覆盖
- **技术**：LLM（GLM-4 / DeepSeek / 通义千问医疗版）+ RAG（药品知识库 + 临床指南）
- **输入**：处方 + 患者上下文（诊断、检验、过敏、既往用药）
- **输出**：自然语言推理结论 + 引用依据
- **风险**：幻觉、推理不可解释、性能成本

### 4.3 规则 + LLM 协同架构

```
处方 + 患者上下文
    ↓
【规则引擎】（必选，覆盖明确规则）
    ├─ ✅ 通过 → 进入调剂
    ├─ ⚠️ 预警 → 药师复审
    └─ ❌ 驳回 → 反馈医生
    ↓
【LLM 推理引擎】（试点，仅处理规则未覆盖的复杂情况）
    ├─ ✅ 推理通过 → 药师最终确认
    ├─ ⚠️ 推理预警 → 药师复审（高优先级）
    └─ ❓ 不确定 → 必转药师复审
    ↓
药师工作站（最终决策权）
```

### 4.4 LLM 接入要点

- **私有化部署**：医疗场景必须私有化部署开源模型（GLM / Qwen / DeepSeek）
- **数据脱敏**：送入 LLM 的患者信息必须脱敏
- **审计追溯**：保留 Prompt、模型输出、药师最终决策
- **不直接驳回**：LLM 输出仅作辅助，最终决策由药师签字
- **成本控制**：仅在规则未覆盖时调用 LLM，避免高频调用

## 六、部署架构

### 6.1 集中式部署（推荐）

```
┌──────────────────────────────────────────────────────────────┐
│ 云审方中心（多租户 SaaS）                                      │
│  - 规则引擎 + 知识库 + 药师工作站 + LLM 推理                  │
├──────────────────────────────────────────────────────────────┤
│ 集成层：FHIR / HIS / EMR / 中心药房 / 互联网医院              │
├──────────────────────────────────────────────────────────────┤
│ 成员单位 1   成员单位 2   ...   成员单位 N                    │
│ HIS/EMR      HIS/EMR      ...   HIS/EMR                       │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 分布式部署

- 每家机构本地化部署审方规则引擎
- 中心统一管理规则库与知识库，定期同步至本地
- 适用于：网络条件差、数据不出院的场景

### 6.3 混合部署

- 规则引擎本地化部署（保证低延迟）
- 药师工作站云化（药师集中办公）
- LLM 推理私有云部署
- 适用于：大型三甲 + 区域集团混合场景

### 6.4 性能与容量

| 指标 | 推荐值 |
|------|--------|
| 规则引擎响应时间 | ≤500ms |
| 药师工作站并发 | 按每药师 100 处方/日 计 |
| LLM 推理响应时间 | ≤5s |
| 系统可用率 | ≥99.9% |
| 规则库规模 | ≥5000 条规则 |
| 处方吞吐量 | 峰值 200 处方/分钟 |

## 八、合规要点

### 8.1 资质合规

- 云审方中心须依托具有《医疗机构执业许可证》的实体机构
- 药师须具备《执业药师资格证》
- 跨机构审方须明确审方药师与开方医生的责任关系

### 8.2 审方规则合规

- 审方规则须经医院药事委员会审核通过
- 规则更新须有审批流程与版本记录
- 严禁绕过审方规则直接调剂（除急救抢救场景，须事后补审）

### 8.3 数据合规

- 审方记录须留存 **≥15 年**（病历保存要求）
- 患者上下文数据传输须加密（TLS 1.2+）
- LLM 推理数据须脱敏
- 跨机构审方须明确数据归属与访问权限

### 8.4 责任边界

- **规则引擎**：辅助工具，不承担医疗责任
- **LLM 推理**：辅助工具，不承担医疗责任
- **审方药师**：最终决策责任，须签字留痕
- **开方医生**：原始处方责任

## 九、常见踩坑

### 9.1 规则误报率高

- **问题**：规则过严导致大量误报，药师疲劳审方
- **应对**：分阶段引入规则、设置灰度期、定期优化规则阈值

### 9.2 规则覆盖不全

- **问题**：罕见药物、特殊人群规则缺失
- **应对**：定期分析"漏审处方"、补充规则、引入 LLM 兜底

### 9.3 药师审方滞后

- **问题**：高峰时段审方排队，影响患者取药
- **应对**：分级审方（普通药师 / 主任药师）、动态调配药师、规则自动通过低风险处方

### 9.4 患者上下文缺失

- **问题**：缺乏诊断、过敏、检验数据，规则无法判断
- **应对**：强制 CDR 对接、缺失数据时降级审核（仅规范审核）

### 9.5 LLM 幻觉

- **问题**：LLM 编造药物相互作用、剂量建议
- **应对**：LLM 仅作辅助、必须药师复核、定期抽检 LLM 准确率

### 9.6 规则冲突

- **问题**：不同来源规则冲突（如药典 vs 临床指南）
- **应对**：建立规则优先级（药典 > 指南 > 厂商说明书）、冲突仲裁机制

### 9.7 跨机构审方责任不清

- **问题**：A 院医生开方、B 院药师审方，责任归属争议
- **应对**：明确集团内审方责任划分协议、保留审方记录留痕

### 9.8 医保规则更新滞后

- **问题**：医保目录与限定支付条件频繁更新，规则库滞后
- **应对**：与医保局接口对接、订阅医保变更通知、定期同步规则库

## 十、与中心药房的协同

云审方中心与中心药房（见 `references/central-pharmacy.md`）通常组合部署：

```
处方上传 → 中心药房接收 → 调用云审方中心审方
                              ↓
                         审方通过 → 中心药房调剂
                         审方预警 → 药师复审 → 通过 / 驳回
                              ↓
                         审方记录归档（双系统共享）
```

**协同要点**：
- 处方流转接口统一（FHIR MedicationRequest 扩展）
- 审方记录双向同步
- 药师工作站与中心药房管理后台共享登录
- 药品主数据共享（避免药品字典不一致导致审方失败）
